该文件是一个基于 SQLite 的本地跑步数据管理模块，核心用于跑步数据的存储、查询和趋势分析，以下是详细总结：
一、核心定位
实现本地跑步数据的持久化存储（SQLite），提供数据增改、查询、趋势统计等核心能力。
二、基础配置与初始化
路径配置：定义数据库文件路径DB_PATH（位于当前目录下data/runs.db），自动创建数据目录。
数据库连接：get_db()函数负责创建 / 连接数据库，启用 WAL 日志模式提升读写性能，设置行工厂为字典格式，并触发表初始化。
表结构初始化：_init_tables()通过 SQL 脚本创建两张核心表：
runs（跑步主表）：主键为label_id，包含运动类型、开始时间、时长、距离、配速、心率、步频、卡路里、爬升 / 下降、功率、训练负荷 / 效果等维度，还存储 JSON 格式的evolab_data（进阶数据）和summary_json（汇总数据），以及数据创建时间。
hr_zones（心率区间表）：关联runs表的label_id，存储心率区间名称、计数、占比，含外键约束；同时为runs表的start_time创建降序索引，优化时间维度查询性能。
三、核心功能函数
表格
函数名	功能描述
save_run	新增 / 替换跑步数据：先写入runs表，删除该label_id下旧的心率区间数据，再插入新心率区间数据，最后提交事务。
get_run	根据label_id查询单条跑步数据，返回字典格式（无数据则返回 None）。
get_recent_runs	查询最近 N 条（默认 30）跑步数据，按start_time降序排列，返回字典列表。
get_trends	统计跑步趋势：取最近 100 条数据，计算总跑量、平均距离 / 配速 / 心率、最佳配速、最长跑步距离，以及按 “年月” 维度汇总的周度跑步次数、总距离、总时长。
四、数据处理特点
复杂数据（evolab_data、summary）通过json.dumps序列化后存储，保证结构化数据的持久化；
数据库查询结果转为字典格式，简化数据读取和使用；
心率区间数据采用 “先删后插” 策略，保证label_id关联数据的一致性。
五、核心场景
支持单条跑步数据的增改查、近期数据列表查询，以及跑步趋势（如配速 / 心率变化、月度跑量汇总）的统计分析，为跑步数据可视化、复盘提供数据支撑。
